## Per-item detail

### P0 — Header icons vanish while dragging the window
**Cause (diagnosed):** semi-transparent header buttons over a continuously-animating body background (`gradientShift` + float pseudo-elements) drop out of the composite during an OS window move under Chromium 106.
**Fix:** promote the leaf buttons (`.icon-btn`, `.status-btn`) to their own compositor layer via `transform: translateZ(0)`.
**Regression caught & fixed:** my first attempt also promoted the `.header-controls` *container*, which created a stacking context that trapped the StatusDropdown menu **below** the message list (the smoke test caught it — the Home option was unclickable). Removed the container promotion; only leaf buttons are promoted now.
**Files:** `ChatWindow.css`, `StatusDropdown.css`.
**Verified:** smoke test — header controls + ≥5 icon buttons render on both roles; StatusDropdown opens and is clickable. Visual drag behavior flagged for your confirmation.

### E1 — 🏠 Home preset status
Added `Home` to `PRESET_STATUSES` (single source of truth); extended the `Status` union; added a one-time migration in `listCustomStatuses()` that drops any saved custom whose label now collides with a preset (persisted quietly). Updated `CLAUDE.md` status list.
**Files:** `custom-statuses.ts`, `ChatWindow.tsx`, `CLAUDE.md`, `custom-statuses.test.ts` (+2 tests).
**Verified:** unit tests (Home preset, emoji, collision migration) + smoke: Home shows in dropdown with 🏠, selecting posts a local system line, updates the button, and the peer's header + system line reflect it.

### E2 — System tray + close-to-tray
Tray icon (embedded base64 PNG — no `electron-builder.yml` change) with **Open / Set status (preset submenu) / Quit**. New Settings toggle "Close button hides to tray" (default OFF), persisted to `app-settings.json` in userData (main-process readable). Close-to-tray hides instead of quitting; a genuine quit is detected via `before-quit` (so tray Quit, `app.quit()`, and the auto-update `quitAndInstall` all still close — **no `updater.ts` change needed**). No single-instance lock added (keeps the two-instance test working).
**Files:** `index.ts`, `tray-icon.ts` (new), `preload/index.ts`, `SettingsMenu.tsx`, `ChatWindow.tsx`.
**Verified:** smoke — with the setting on, `window.close()` hides (not destroys) the window; tray "Set status" → renderer applies the status.

### E3 — Quiet hours (scheduled DND)
New pure util `quiet-hours.ts` (`isQuietNow` handles overnight wraparound + zero-length windows). Header 🌙 indicator (30s poll + change event). Settings section with enable toggle + time pickers.
**Files:** `quiet-hours.ts` (new), `ChatWindow.tsx`, `ChatWindow.css`, `SettingsMenu.tsx`, `quiet-hours.test.ts` (new, 9 tests).
**Verified:** unit tests (overnight, same-day, boundaries, malformed, persistence) + smoke: toggle + time pickers present, 🌙 shows when active, messages still arrive during quiet hours.

### E4 — Connection-quality indicator
Surfaces the already-collected `lastRttMs` via a 5s `getDiagnostics()` poll. **Plan assumption corrected:** RTT is measured **only on the connector (client)** — the hub/listener's diagnostics have no `lastRttMs`. So a single global indicator renders only when a numeric RTT exists (connectors), never on the hub. Green <100ms / yellow <300ms / red ≥300ms with a "Ping: N ms" tooltip. No new protocol traffic.
**Files:** `ChatWindow.tsx`, `ChatWindow.css`.
**Verified:** smoke — connector shows `.rtt-indicator` with a ms value + quality class; hub shows none.

### E5 — Export chat history
New `history:export` IPC reusing the existing decrypt path; renders **HTML** (sender-colored, day headers, `[Photo]`/`[File]`/`[Voice message]` placeholders, HTML-escaped) or **plain .txt**, chosen by the save-dialog file type. Rendering extracted to a pure module for testability. **Deviation:** image *embedding* (the plan's optional data-URI checkbox) was not implemented — placeholders only — to stay in scope and avoid data-URI bloat; noted as a deliberate deferral.
**Files:** `handlers.ts`, `history-export.ts` (new), `preload/index.ts`, `SettingsMenu.tsx`, `history-export.test.ts` (new, 10 tests).
**Verified:** unit tests (format, escaping, placeholders, day headers, empty) + smoke: with the save dialog stubbed, a real valid HTML file is written and validated on disk.

### E6 — "New messages" divider + jump-to-unread
`firstUnreadId` captured when a message arrives while the window is unfocused; a "New messages" divider renders above it (sibling node, keyed by id — memoization-safe). A "↓ N new" pill appears when scrolled up and jumps to the divider (expanding the render window first if it's above the slice). Divider persists on refocus, retires on blur; in-memory only (no stale divider after restart).
**Files:** `ChatWindow.tsx`, `ChatWindow.css`.
**Verified:** smoke (focus simulated via `document.hasFocus` override) — divider appears above the first unread, persists on refocus, the pill appears when scrolled up and jumps, divider retires on blur.

### E7 — Pinned messages
`pinned?: boolean` on `Message`; 📌 pin/unpin action in both message toolbars; new `pin`/`unpin` protocol types added to the type union and to the hub `RELAY_TYPES`; incoming handler toggles `pinned`. A slim pinned bar under the header cycles pinned messages (jump / unpin / N-of-M cycle). Pins ride in history (`history:save` keeps all fields; history-sync passes whole objects) so they persist and sync.
**Protocol compat (verified in code first, per the mandatory rule):** the client `default` case and the server's post-switch `emit('message')` both forward unknown types generically; the renderer's if/else ignores unhandled types — so a 3.4.x peer receiving `pin`/`unpin` silently ignores it and does not crash.
**Files:** `protocol.ts`, `tcp-server.ts`, `ChatWindow.tsx`, `ChatWindow.css`, `MessageBubble.tsx`.
**Verified:** smoke — pin shows the bar, syncs to the peer, unpin syncs and clears on both sides; and an unknown message type (`pin-future-variant`) is ignored with chat still flowing (the simulated-old-client check).

### E8 — Per-message read-aloud
🔊 action on text bubbles → direct `ttsSpeak` (stable `useCallback`, memoization-safe). Skips system + empty messages. Respects mute and quiet hours; guarded to no-op during an active auto-read/voice flow so it never triggers the mic auto-open.
**Files:** `MessageBubble.tsx`, `ChatWindow.tsx`.
**Verified:** smoke — 🔊 appears on a text message, absent on system messages. Audio flagged for your confirmation.

### E9 — Spoken announcements
Default-OFF Settings toggle (`announce-prefs.ts`). When on and in a speech status, TTS announces peer status changes (only genuine ones — hooked after the existing changed-check), incoming calls, and reconnects. Speaks directly (no mic auto-open); skips while a message is being read or during mute/quiet-hours; no await-before-speak race.
**Files:** `announce-prefs.ts` (new), `ChatWindow.tsx`, `SettingsMenu.tsx`, `announce-prefs.test.ts` (new, 3 tests).
**Verified:** unit tests + smoke: toggle present, defaults OFF, persists ON. Audio flagged for your confirmation.

### E10 — Shared media gallery
New `MediaGallery.tsx` (ReleaseNotes modal pattern — overlay + Esc). Reads the **full** `messages` array (not the windowed slice) via `useMemo`: images as a lazy-loaded thumbnail grid (click → existing lightbox), other files/voice as a newest-first list. 🖼️ header button opens it. No new storage/protocol.
**Files:** `MediaGallery.tsx` + `.css` (new), `ChatWindow.tsx`.
**Verified:** smoke — after sending a real image peer-to-peer, the gallery lists it as a thumbnail; closes via ✕ and Esc.
